# Code Review Skill

## 能力
审查代码质量，发现问题并给出结构化审查报告。

## 何时使用
- 用户要求 review 代码
- 提交 PR 前
- 代码重构后
- 输入包含 "review"、"审查"、"代码质量" 等关键词

## 审查工作流程
1. **收集变更**：读取变更文件列表和 diff 内容
2. **理解上下文**：读取相关文件的完整代码，理解改动意图
3. **逐维度审查**：
   - **正确性**：逻辑漏洞、边界情况、类型安全、异常处理
   - **安全性**：注入攻击、硬编码密钥、不安全的 API 调用、权限绕过
   - **性能**：N+1 查询、内存泄漏、不必要的同步操作、大循环优化
   - **可维护性**：命名规范、函数长度、圈复杂度、注释质量
   - **最佳实践**：SOLID 原则、DRY、设计模式、错误处理模式
4. **生成报告**：按严重程度分级，每个问题附带修复建议和代码示例

## 输出格式

```markdown
## 审查报告

### 🔴 严重问题（必须修复）
- `[文件:行号]` 问题描述
  - **修复建议**: 具体修改方式
  - **代码示例**: 修复后的代码片段

### 🟡 建议优化（推荐修复）
- `[文件:行号]` 优化建议

### 🟢 做得好
- 值得肯定的编码实践

### 📊 总结
- 审查文件数：X
- 严重问题：X | 建议：X
- 整体评分：X/10
```

## 审查规则

### 安全红线（必须报告）
- 硬编码密钥或 token
- SQL 注入 / XSS / CSRF 风险
- 不安全的反序列化
- 文件路径遍历
- 不安全的正则（ReDoS）

### 性能关注点
- 同步文件 I/O
- 未分页的数据库查询
- 循环内的 await
- 闭包导致的内存泄漏风险

### 代码规范
- 函数不超过 50 行
- 圈复杂度不超过 10
- 嵌套不超过 4 层
- 参数不超过 5 个

## 配置项
- `strictness`: `"loose"` | `"normal"` | `"strict"` — 审查严格程度
- `lintRules`: `string[]` — 额外的 lint 规则
