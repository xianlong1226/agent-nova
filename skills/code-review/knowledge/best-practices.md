# 代码审查最佳实践

## 通用准则
1. 先理解意图，再评判实现
2. 关注接口契约，而非实现细节
3. 区分"必须修复"和"个人偏好"
4. 每条意见都附带改进建议

## TypeScript 专项
- 避免 `any`，使用 `unknown` 或具体类型
- 使用 `readonly` 标记不可变数据
- 优先 `interface` 扩展，而非 `type` 交叉
- 枚举使用 `as const` 对象替代 `enum`
- 使用类型守卫收窄联合类型

## React 专项
- 组件文件不超过 200 行
- 自定义 Hook 抽离副作用逻辑
- 使用 `useCallback` / `useMemo` 避免不必要的重渲染
- Props 定义独立 interface，不内联
- 列表渲染必须指定稳定的 key

## Node.js 专项
- 异步操作统一用 async/await
- 错误处理用 try/catch，不留空 catch
- 流操作注意背压控制
- 注意事件监听器泄漏
- 进程退出前清理资源

## 安全审查清单
- [ ] 所有外部输入已校验和清洗
- [ ] 无硬编码密钥或凭据
- [ ] SQL 使用参数化查询
- [ ] HTML 输出已转义
- [ ] 文件路径已标准化（防路径遍历）
- [ ] 正则表达式不会触发 ReDoS
- [ ] 依赖包版本锁定
